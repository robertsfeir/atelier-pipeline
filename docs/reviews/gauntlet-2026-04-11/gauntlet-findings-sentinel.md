# Sentinel — Security Audit

**Scope:** atelier-pipeline full codebase (brain/ Node.js MCP+HTTP server, brain/ui/ vanilla JS dashboard, source/claude/hooks/ Bash enforcement scripts).
**Mode:** READ-ONLY, Semgrep-backed SAST + manual cross-reference against diff-free audit.
**Date:** 2026-04-11
**Reviewer:** Sentinel

---

## Summary

The audit surfaces **one critical exposure**, **three high-severity issues**, and a cluster of medium/low findings concentrated in the HTTP server's security control composition. The brain HTTP server correctly implements a Bearer-token auth middleware for `/api/*` routes and parameterizes every SQL query — but the same server exposes the entire MCP tool surface (`agent_capture`, `atelier_hydrate`, `agent_search`, ...) on the HTTP transport with **no authentication check**. An attacker who can reach `localhost:8788` (malicious script in any browser tab, any local user on shared machines, any container sidecar) can invoke every brain tool, read and write all thoughts, and trigger the LLM/embedding outbound calls that burn the OpenRouter budget.

The auth control is **non-composable**: `checkAuth` lives inside `rest-api.mjs` and is applied only to `/api/*`. The MCP path skips it entirely. A second instance of the same composition defect: `server.mjs:114` sets a per-response CORS origin of `http://localhost:${PORT}`, and four telemetry handlers in `rest-api.mjs` *override* that with `Access-Control-Allow-Origin: *`. The telemetry endpoints are auth-gated, so this is not directly exploitable as data exfiltration (preflight with the Authorization header will still be blocked by the server's own CORS response — no wildcard credential allowance is sent), but the inconsistency is a latent trap: add one credentialed fetch pattern to those routes and you have a cross-origin read.

The dashboard injects the API token into the HTML via `window.__ATELIER_API_TOKEN__` — any XSS in the dashboard UI is immediately a token exfiltration. Dashboard rendering mixes `innerHTML` string concatenation with an `escapeHtml` helper that is applied inconsistently (line 1163 of `dashboard.html` renders `s.sub2` raw where all sibling fields are escaped). Today the source data for that one field is numeric/trusted, but the pattern is fragile. A Content Security Policy header is absent on every response; the dashboard loads `chart.js` from `cdn.jsdelivr.net` with **no Subresource Integrity attribute**, so CDN compromise = total XSS.

On the Bash hooks, the shell scripts follow defensive patterns (`set -euo pipefail` where appropriate, `|| true` for non-critical I/O, `jq` for JSON parsing rather than regex, `grep -qE` with anchored patterns). No command injection was found. The ADR-0031 permission audit trail is well-designed: enforcement hooks log JSONL locally with no network path, and hydration is decoupled and fail-open. One minor hardening opportunity: `echo "$INPUT" | jq` relies on `echo`'s handling of backslash sequences, which varies by shell. `printf '%s' "$INPUT" | jq` would be more defensive.

Scan coverage: **15 files scanned by Semgrep across two scans** (7 brain/lib + 8 hooks/ui). Semgrep surfaced **5 findings** directly (4 path-traversal variants in `static.mjs`, 1 missing SRI in `dashboard.html`). The remaining findings in this report come from manual cross-reference against the Semgrep output and the audit scope.

## Positive Observations

1. **All SQL is parameterized.** Every `pool.query` / `client.query` call in `brain/lib/rest-api.mjs`, `brain/lib/tools.mjs`, `brain/lib/conflict.mjs` uses pg's `$1..$N` parameter binding. The `handlePutConfig` dynamic UPDATE builder at `rest-api.mjs:141-158` enforces an explicit column allowlist before constructing the SET clause, and then still binds values via parameters — the column names are the only interpolated tokens and are checked against a static list. This is textbook correct, and deserves explicit credit because it is the single most common vulnerability class in Node HTTP apps. CWE-89 coverage: strong.

2. **Config field validation + allowlist at the edge.** `rest-api.mjs:132-177` defines `validateConfigField` which enforces type and range constraints for every config field (`interval/min/max` must be non-negative number, `threshold` must be 0-1). Unknown fields are silently dropped rather than accepted. The allowlist is hard-coded in the handler, not data-driven, so it cannot be bypassed by polluting the DB. This is defense-in-depth over the DB enum constraints.

3. **Enforcement path is decoupled from network I/O.** ADR-0031's permission audit trail design is exemplary: each enforcement hook appends one line to `~/.claude/logs/enforcement-YYYY-MM-DD.jsonl` via `printf >> ... 2>/dev/null || true`. No hook makes a network call, no hook depends on the brain being up, no hook retries on failure. Hydration is a separate SessionStop script that bulk-captures blocked events with fail-open semantics. This decoupling means enforcement never becomes a DoS vector or a latency source in the hot path — a common failure mode in audit-logging designs.

4. **Crash guards and graceful shutdown.** `server.mjs:71-76` installs `installCrashGuards` that wires `process.exit`, the consolidation/TTL timers, and the pool end into a coordinated shutdown. Uncaught exceptions should therefore close DB connections cleanly rather than leak them. (Not audited in depth; worth verifying the `crash-guards.mjs` implementation catches both `uncaughtException` and `unhandledRejection`.)

5. **Retry policy on external API calls is bounded and non-recursive.** `embed.mjs:12-90` caps retries at 3, uses fixed exponential backoff `[1000, 2000, 4000]`, only retries on 5xx/429/network errors, and re-throws on 4xx. No infinite loops, no retry on auth failures. This is a direct implementation of retro-lesson #004 and is worth preserving.

6. **Body size cap on REST API.** `rest-api.mjs:439-457` enforces a 1MB request body size limit and destroys the request on overflow. This protects the authenticated REST surface from memory-exhaustion attacks via oversized PUT payloads. (Gap: the same cap does not apply to the MCP path — see Finding #8.)

7. **Hook input parsing uses `jq`, not regex or `eval`.** Every enforcement hook in `source/claude/hooks/enforce-*.sh` pipes stdin through `jq -r '.field // empty'`. There is no `eval`, no `source`, no dynamic command construction from hook input. Shell injection via the hook input channel is architecturally foreclosed. The fallback regex parsers in `log-stop-failure.sh:36-38` and `log-agent-start.sh:46-48` are only used when `jq` is absent and still avoid `eval` by using `sed` with literal patterns.
