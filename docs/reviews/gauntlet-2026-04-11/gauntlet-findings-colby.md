# Colby — Implementation Quality & Technical Debt

**Reviewer:** Colby (Senior Engineer)
**Date:** 2026-04-11
**Scope:** Full atelier-pipeline codebase — brain/lib, brain/server.mjs, brain/scripts, source/claude/hooks, .claude/agents
**Mode:** Read-only audit. No files modified.

## Summary

The brain MCP server is well-modularized (ADR-0003 modularization holds up — clean separation between `db`, `tools`, `rest-api`, `conflict`, `consolidation`, `ttl`, `static`). Transaction discipline in `agent_capture` is correct. Union-find clustering in consolidation is genuinely good.

That said, there is one **Critical** correctness bug that silently voids the entire brain-extractor data pipeline for 6 of 15 agents: `brain/lib/config.mjs` SOURCE_AGENTS and SOURCE_PHASES Zod enums are stale relative to what `.claude/agents/brain-extractor.md` instructs the extractor to emit. Every `agent_capture` call for `sentinel`, `darwin`, `deps`, `brain-extractor`, `robert-spec`, `sable-ux` is rejected by Zod validation before reaching the DB. Every `source_phase: 'product'`, `'ux'`, or `'commit'` — which the mapping table in the extractor agent file explicitly emits — is likewise rejected. The captures disappear with no visible error to the user because the extractor's constraint says "log and continue" on capture errors, and the finally receipt line still claims `{N} captures` based on the extractor's internal count, not DB confirmation.

Secondary themes: (1) configuration drift between three enum sources of truth (Zod enums in `config.mjs`, Postgres enums via migrations, and hand-edited agent persona files) with no cross-check; (2) technical debt in `db.mjs` — 7 copy/paste migration blocks with no `schema_migrations` tracking table, idempotency-by-inspection patterns that diverge per migration; (3) inline validation in REST API layer while MCP tool layer uses Zod (divergent discipline across layers); (4) module-level mutable state (`brainConfigCache`, `consolidationTimer`, `hydrateStatusMap`) is not race-safe in HTTP mode where multiple concurrent MCP sessions exist; (5) hardcoded magic values: LLM model strings (`openai/gpt-4o-mini`) appear in two separate call sites with no shared constant, CORS origin hardcoded to `localhost:${PORT}` at server.mjs:114; (6) fire-and-forget `setImmediate` in `atelier_hydrate` swallows errors that the caller cannot see unless they poll `atelier_hydrate_status`.

The good news: no TODO/FIXME/HACK markers in production code. No commented-out dead code. Crash-guards module is excellently designed (injectable deps, re-entry guard). Body-size limit on REST API readBody is present. Error transactions roll back correctly.

## Positive Observations

1. **Modularization holds up (ADR-0003).** The `brain/lib/` split is clean and consistent — each module has a clear dependency boundary and a single responsibility. `tools.mjs` registers tools via one-function-per-tool pattern, keeping cyclomatic complexity low. `registerAgentCapture` → `handleAgentCapture` → `handleMerge` / `insertThought` / `handleRelations` is a textbook refactor for keeping tool handlers readable. Importing downstream modules in `server.mjs` is a clean list.

2. **Transaction discipline in `agent_capture` is correct.** `handleAgentCapture` at tools.mjs:88-166 uses explicit `BEGIN`, `COMMIT`/`ROLLBACK` on every exit path, and the outer try/finally releases the client unconditionally. The rollback on embedding failure (line 100) is correct — you don't want a half-built thought. The conflict-path has a minor boundary concern (finding 14) but the happy path is solid.

3. **Union-find in `consolidation.mjs` is genuinely good.** `createUnionFind` (lines 15-57) with path compression and union-by-rank is the right algorithm for clustering similar thought pairs. The single-query `SIMILARITY_PAIRS_SQL` (lines 63-81) that lets Postgres compute all pairwise cosine distances in-database, then doing the union-find in JS, is an excellent decomposition — it pushes the expensive vector math to pgvector and keeps the cluster-formation logic simple in-process.

4. **Crash-guards module is excellently designed.** Injectable dependencies (`exitFn`, `stopConsolidation`, `stopTTL`, `poolEnd`) make the module test-friendly and decoupled. The re-entry guard via closure prevents double-cleanup on rapid-fire signals. This is the right shape for a process-lifecycle concern, even though it has one wart around `removeAllListeners` (finding 13).

5. **REST API body-size guard.** `readBody` at rest-api.mjs:439-457 enforces a 1 MB cap via `req.destroy()` on overflow. This is proactive DoS protection that a lot of Node HTTP handlers forget. It's a shame the same rigor was not applied to the MCP string inputs (finding 17).

6. **Batch superseded_by lookup in `atelier_trace`.** The ADR-0026 comment at tools.mjs:537-551 correctly notes that the previous per-node loop was replaced with a single `array_agg` query. This is the right kind of optimization — not premature, not cargo-culted, measurably better because chain sizes are bounded but >1.

7. **Cycle detection in `atelier_relation`.** `checkSupersedeCycle` (tools.mjs:485-498) uses a depth-limited recursive CTE (`chain.depth < 20`) to prevent infinite recursion on pathological chains. The cycle check happens inside the same transaction as the insert, so there is no TOCTOU race. Clean.

8. **No TODO/FIXME/HACK markers in production code.** Scanned across `brain/` and `source/`. No unfinished markers, no commented-out dead code blocks. This is rare and worth calling out — it means the codebase respects its own "no unfinished work" discipline.

9. **Error propagation pattern is consistent.** Every MCP tool returns `{ content: [{ type: "text", text: \`Error: ${err.message}\` }], isError: true }` on failure. Same shape everywhere. No mixed try/catch styles. This is the kind of consistency that makes the codebase easy to reason about — a reader can tell what's going on from any one handler.
