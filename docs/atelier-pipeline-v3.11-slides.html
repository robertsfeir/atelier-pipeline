The Problem

AI writes confident code. Not correct code.

LLMs generate code that compiles, passes a skim, and looks reasonable. Underneath:

Failure Modes
Hallucinated APIs — calling methods that don't exist in your codebase
Pattern drift — inventing a new pattern when one exists 3 files away
Missing state handling — happy path works, error/empty/overflow forgotten
Security blindspots — no auth check, no validation, data in errors
Structural Problems
No memory — every session starts from zero context
No adversarial review — the builder rubber-stamps itself
No session recovery — crash mid-feature and start over
No learning — the 50th run makes the same mistakes as the 1st
Takeaway

The question isn't whether AI can write code — it's whether your system catches mistakes faster than your users do.

The Orchestrator

Eva: the central nervous system

Without a dedicated orchestrator, the main thread is both designing and routing. That creates three problems:

Context Pollution

The main thread accumulates every turn. By step 4, it forgets step 1. Eva externalizes state to disk.

No Separation

When the thread is both designing and routing, concerns compete. Eva handles traffic so agents think deeply.

No Recovery

Claude Code crashes mid-pipeline? Eva reads pipeline-state.md and resumes from the exact step.

DomainResponsibility
Traffic controlRoutes between agents, manages phase transitions, queues fixes back to Colby
State management5 files on disk: pipeline state, context brief, error patterns, investigation ledger, last QA report
Quality & learningTracks error categories. Recurring patterns (3+) become warnings injected into agent prompts
Phase sizingMicro / Small / Medium / Large — a bug fix doesn't need 12 agents
Brain integrationPre-fetches brain context, injects into agent prompts, captures knowledge from agent output
Takeaway

Eva never writes code. She reads, routes, and tracks state. Separate orchestration from execution and both get better.

Institutional Memory

The Brain: not what changed, but why

Git tells you what changed. Commit messages tell you what was done. But why was Option A chosen over Option B? Why did the user reject that approach? That knowledge dies when the terminal closes.

Takeaway

"The agent that fixed your auth bug last week remembers why it chose that approach — even though it's a completely new session."

How It Works

Three-axis scoring: how the brain decides what matters

Every retrieval ranks thoughts by a weighted formula. Relevance dominates. Importance separates decisions from noise. Recency is a tiebreaker.

score = 3.0 × similarity + 2.0 × importance + 0.5 × recency
similarity = cosine(query_embedding, thought_embedding)  •  recency = 0.995hours_since_access  •  importance = 0–1 per thought type

Relevance  w=3.0

Cosine similarity between your query and every stored thought (1536-dim vectors via text-embedding-3-small). What you're asking about matters most.

Importance  w=2.0

Set at capture time per thought type. Preferences = 1.0, decisions = 0.9, lessons = 0.7, rejections = 0.5. Decisions outrank tactical findings.

Recency  w=0.5

Exponential decay from last access time. Half-life ~6 days. Lowest weight on purpose — a 3-month-old decision still surfaces if it's relevant.

Conflict Detection (on write)
Similarity > 0.9 — duplicate detected, merged into existing thought
Similarity 0.7–0.9 — LLM classifies: DUPLICATE / CONTRADICTION / SUPERSESSION / COMPLEMENT / NOVEL
Similarity < 0.7 — novel thought, stored directly
Contradictions in same scope → auto-supersede. Cross-scope → flagged as conflicted
Lifecycle (background)
TTL enforcement (hourly) — expires thoughts past their type's TTL. Decisions: never. Drift findings: 90 days
Consolidation (every 30 min) — clusters similar thoughts via union-find, synthesizes into higher-level reflections
Relation graph — supersedes, triggered_by, evolves_from, contradicts, supports, synthesized_from
Access tracking — every retrieval updates last_accessed_at, feeding the recency axis
Takeaway

The brain doesn't just store everything — it deduplicates, detects contradictions, expires stale knowledge, and consolidates clusters into higher-level insights. Automatically.

Testing

Contract-first TDD: Roz writes tests. Colby makes them pass.

The tester and builder are different agents. The tester defines the contract before any implementation exists.

Roz — QA Engineer
Reviews Cal's test spec for gaps
Writes acceptance tests from the spec
All tests FAIL until Colby implements
Adversarial: failure tests ≥ happy path
Write access: test files ONLY
Colby — Sr. Engineer
Reads Roz's tests as the contract
Implements to make acceptance tests pass
Adds unit tests for implementation details
Cannot modify Roz's acceptance test files
Self-validates: lint, typecheck, drift check
Takeaway

The test becomes the contract, not a rubber stamp. When one agent writes the tests and another writes the code, the tests can't just agree with the implementation.

Quality

Continuous QA: build one, review one

We stopped reviewing code in batches. Eva groups independent build steps into waves and interleaves build with review.

Before: Batch Review

Build steps 1–6. Review all. Drift in step 2 corrupts everything after. Six steps thrown out.

After: Interleaved
Eva→Colby: Step 1 → Roz: QA ✓
Eva→Colby: Step 2 → Roz: QA ✗
  └ Eva queues fix → Colby → Roz ✓
Eva→Colby: Step 3 → Roz: QA ✓
Eva→Roz: Final full sweep
Eva→Ellis: Commit

Wave Grouping

Eva extracts file dependencies, groups independent steps into waves. Waves run sequentially; units within a wave can run in parallel.

Scoped Re-runs

Fix cycles only re-run failed checks + affected tests. 60–70% fewer tokens per cycle. No full re-review.

Review Juncture

After all waves pass: Roz final sweep + Poirot blind review + Robert verify + Sable verify + Sentinel (if enabled) — all in parallel.

Takeaway

Drift caught where it starts. Fix queues are small. Colby → Roz → Ellis is the minimum pipeline. No sizing level skips Roz or Ellis.

What's New — v3.9+

Darwin: the pipeline that improves itself

Darwin analyzes telemetry, evaluates agent fitness, and proposes structural improvements. Then tracks whether its own edits helped or caused regressions.

What Darwin Does
Analyzes telemetry across pipeline runs — cost, duration, rework rate, QA pass rate
Evaluates agent fitness — which agents are underperforming?
Proposes structural edits — persona changes, constraint adjustments, flow optimizations
Tracks its own impact — did the edit improve the target metric?
Self-Correction Loop
Baseline captured before each edit is applied
3+ subsequent pipelines required before measuring delta
Improvement logged: "Darwin edit #4: rework rate improved 2.1 → 1.4"
Regression flagged: "Warning: edit #7 may have caused regression. Consider reverting."

Darwin edits require user approval. Read-only access to the codebase. Proposals, not autonomous changes.

Takeaway

The pipeline evolves without manual tuning. Darwin's self-correction loop means bad edits get flagged and good edits get reinforced — automatically.

Enforcement

Behavioral rules are nice. Shell scripts that block tool calls are better.

PreToolUse hooks intercept every Write, Edit, and Bash call. They reject tool calls that violate agent boundaries — before the tool executes.

What Gets Enforced
Eva — cannot Write or Edit any file outside docs/pipeline/
Roz — can only write to test files. Cannot touch implementation code
Ellis — blocked during active pipeline until Roz has passed QA
Colby — blocked from paths specified in config (e.g., infra files)
Read-only agents (Poirot, Robert, Sable, Sentinel, Deps, Darwin) — Write/Edit/MultiEdit blocked entirely
How It Works
3 hook scripts in .claude/hooks/ + 1 config
enforce-paths.sh — blocks writes outside agent's lane
enforce-sequencing.sh — blocks agents from acting out of turn
enforce-quality-gate.sh — runs lint + tests on Stop
Result: a rejected tool call, not a polite suggestion to try again
Takeaway

Prompt instructions say "don't do this." Hooks say "you can't do this." The difference matters when agents are creative about workarounds.

What's New — Observability

Telemetry: see what the pipeline is doing

Every pipeline run captures agent-level telemetry. The dashboard shows trends. Eva reads degradation alerts at session boot.

Per-Agent Metrics

Cost, duration, token usage, model used, rework cycles. Captured automatically after each agent returns.

Pipeline Trends

Day-aggregated cost trends. Rework rate over time. First-pass QA rate. EvoScore (Darwin fitness). Honest metrics — no vanity numbers.

Degradation Alerts

Fire when thresholds are exceeded for 3+ consecutive pipelines. Cost trending up >25%. Rework rate >2.0. First-pass QA <60%.

CI Watch

Self-healing CI protocol. Monitors CI pipeline runs, detects failures, classifies the failure type, and routes fixes automatically. Configurable max retries. The CI doesn't just report red — it fixes red.

Takeaway

You can't improve what you can't measure. Telemetry + Darwin create a feedback loop: measure, propose, apply, re-measure.

Takeaways

Six things to remember

1

Structure beats power.

A well-designed workflow with Sonnet outperforms ad-hoc Opus prompting. The magic isn't the model — it's the roles, gates, and handoffs.

2

Separate who writes tests from who writes code.

When the tester authors tests before the builder starts, tests become a contract — not the code agreeing with itself.

3

Enforce mechanically, not behaviorally.

Shell scripts that reject tool calls beat prompt instructions that agents creatively work around. Trust but verify — with code.

4

Build memory that outlives the session.

Context briefs, error patterns, and a persistent brain mean the 50th pipeline run is harder to fool than the 5th.

5

Make QA continuous, not batch.

Interleave build and review. Catch drift where it starts. Keep fix cycles small. The minimum pipeline is always Colby → Roz → Ellis.

6

Let the pipeline improve itself.

Telemetry + Darwin create a feedback loop. Measure agent performance, propose improvements, track whether edits helped. Self-evolving infrastructure.
